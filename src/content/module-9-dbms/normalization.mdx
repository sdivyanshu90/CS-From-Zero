## Hook — Why Your Data Is Lying to You

A table stores student name, department name, and department head in every row. The department head changes — now you must update hundreds of rows. Miss one? Inconsistent data. This is an **update anomaly**, and **normalization** systematically eliminates these problems.

## The Anomalies

```
StudentCourse table (unnormalized):
| sid | sname  | dept    | dept_head | course | grade |
|-----|--------|---------|-----------|--------|-------|
| 1   | Alice  | CS      | Dr. Smith | DB     | A     |
| 1   | Alice  | CS      | Dr. Smith | OS     | B     |
| 2   | Bob    | Math    | Dr. Jones | Calc   | A     |

Update Anomaly: Change dept_head for CS → must update ALL CS rows.
Insert Anomaly: Can't add new department without a student.
Delete Anomaly: Delete Bob → lose Math department info entirely!
```

## Normal Forms

### First Normal Form (1NF)

```
Rule: Every column must contain atomic (single) values.
      No repeating groups or arrays.

Violation:
| sid | courses          |
| 1   | DB, OS, Networks |   ← NOT atomic!

Fix: One row per course.
| sid | course   |
| 1   | DB       |
| 1   | OS       |
| 1   | Networks |
```

### Second Normal Form (2NF)

```
Rule: 1NF + no partial dependencies.
      Every non-key attribute depends on the ENTIRE primary key.

Table: Enrollment(student_id, course_id, student_name, grade)
PK: (student_id, course_id)

student_name depends on student_id alone (partial dependency!)
grade depends on (student_id, course_id) — full dependency ✓

Fix: Split into:
  Student(student_id, student_name)
  Enrollment(student_id, course_id, grade)
```

### Third Normal Form (3NF)

```
Rule: 2NF + no transitive dependencies.
      Non-key attributes must not depend on other non-key attributes.

Table: Student(sid, sname, dept_id, dept_name, dept_head)

dept_name depends on dept_id (not on sid directly)
sid → dept_id → dept_name (transitive!)

Fix: Split into:
  Student(sid, sname, dept_id)
  Department(dept_id, dept_name, dept_head)
```

### Boyce-Codd Normal Form (BCNF)

```
Rule: Every determinant must be a candidate key.
      Stricter than 3NF.

3NF allows: A → B where A is part of a candidate key.
BCNF forbids this: only full candidate keys can determine anything.

In practice, 3NF is usually sufficient.
Most real databases are designed to 3NF.
```

## How It Actually Works — Denormalization

```
Normalization: Split tables to eliminate redundancy.
  + No anomalies, less storage, easier updates
  - More JOINs needed for queries → slower reads

Denormalization: Intentionally add redundancy back.
  + Faster reads (fewer JOINs)
  - Risk of anomalies, more storage, harder updates

Real systems: Normalize first, then denormalize specific
tables for performance where needed. Measure, don't guess.
```

## Why It Matters

- Normalization prevents data corruption from update/insert/delete anomalies.
- Understanding normal forms is essential for database design interviews and real projects.
- Knowing WHEN to denormalize (read-heavy analytics) vs. normalize (write-heavy OLTP) is a key architectural decision.
