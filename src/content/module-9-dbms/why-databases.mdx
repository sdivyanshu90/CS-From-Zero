## Hook — Why Not Just Use Files?

You could store all your data in text files. But what happens when two users edit the same file simultaneously? What if the power fails mid-write? What if you need to find one record among millions? **Databases** solve these problems with concurrency, durability, and efficient querying — the backbone of every modern application.

## The Problems with Files

```
Problem 1: Data Redundancy
  Student name stored in enrollment file, grade file, contact file...
  Update name in one → inconsistent data everywhere.

Problem 2: No Concurrency Control
  Two users update the same file → one overwrites the other.
  No isolation between operations.

Problem 3: No Atomicity
  Transfer $100: debit account A, credit account B.
  Crash after debit → $100 vanishes!

Problem 4: No Query Language
  "Find all students with GPA > 3.5 who enrolled after 2020"
  → Write custom parsing code every time.
```

## What Is a DBMS?

```
Application → SQL Query → DBMS → Stored Data
                          │
                ┌─────────┴──────────┐
                │  Query Processor    │
                │  Transaction Mgr   │
                │  Buffer Manager    │
                │  Storage Manager   │
                │  Recovery Manager  │
                └────────────────────┘

A DBMS provides:
  1. Data Definition:  CREATE TABLE, schemas
  2. Data Manipulation: INSERT, SELECT, UPDATE, DELETE
  3. Data Control:     GRANT, REVOKE (access control)
  4. Transaction Mgmt: ACID properties
  5. Concurrency:      Multiple users simultaneously
  6. Recovery:         Survive crashes without data loss
```

## See It Work — ACID Properties

```
Atomicity:    All or nothing. Transfer either fully completes
              or fully rolls back. Never half-done.

Consistency:  Database moves from one valid state to another.
              Constraints (foreign keys, checks) always hold.

Isolation:    Concurrent transactions don't interfere.
              Each sees a consistent snapshot.

Durability:   Once committed, data survives any failure
              (power loss, crashes). Written to disk + log.
```

## How It Actually Works — Architecture

```
                    ┌──────────────┐
  SQL Query ───────→│ Parser       │ Parse SQL into tree
                    │ Optimizer    │ Choose best execution plan
                    │ Executor     │ Execute the plan
                    ├──────────────┤
                    │ Buffer Pool  │ Cache disk pages in RAM
                    │ Lock Manager │ Concurrency control
                    │ Log Manager  │ Write-ahead log for recovery
                    ├──────────────┤
                    │ Disk Storage │ Data files, index files, logs
                    └──────────────┘
```

## Why It Matters

- Every application you use (social media, banking, email) is backed by a database.
- Understanding DBMS design explains why queries are fast or slow, and how data stays safe.
- Database skills (SQL, schema design, optimization) are among the most valuable in tech.
