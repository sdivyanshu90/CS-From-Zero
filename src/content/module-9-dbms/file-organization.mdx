## Hook — How Data Lives on Disk

A database table isn't a spreadsheet in memory — it's blocks of bytes on a spinning disk or SSD. How those bytes are organized determines whether your query takes 1 millisecond or 10 minutes. **File organization** is the physical foundation beneath SQL's logical abstraction.

## Records and Blocks

```
Disk → divided into blocks (pages), typically 4-8 KB
Each block holds multiple records (rows).

Block (4KB):
┌──────────────────────────────────────┐
│ Header (free space pointer, count)    │
│ Record 1: [id=1, name="Alice", ...]  │
│ Record 2: [id=2, name="Bob", ...]    │
│ Record 3: [id=3, name="Carol", ...]  │
│ ... free space ...                    │
└──────────────────────────────────────┘

Fixed-length records: Easy to locate (record i at offset i × size)
Variable-length records: Need slot directory (pointers to each record)
```

## See It Work — Organization Methods

### Heap File (Unordered)

```
Records stored in insertion order. No sorting.

Insert: O(1) — append to end.
Search: O(n) — must scan entire file.
Delete: Mark as deleted, periodically compact.

Good for: Bulk loading, tables always scanned fully.
Bad for: Point queries without an index.
```

### Sorted File (Sequential)

```
Records sorted by a key (e.g., id).

Search: O(log n) — binary search on blocks.
Insert: O(n) — must shift records to maintain order!
Range query: Excellent — sequential read of contiguous blocks.

Good for: Range queries on sort key.
Bad for: Frequent inserts (shifting is expensive).
```

### Hash File

```
Hash function maps key to bucket (block number).
bucket = hash(key) mod num_buckets

Search: O(1) — one disk read (if no overflow).
Insert: O(1) — hash and append to bucket.
Range query: Terrible — hashing destroys order.

Overflow handling:
  Chaining: Overflow bucket linked to main bucket.
  Extendible hashing: Directory doubles when needed.
  Linear hashing: Buckets split incrementally.
```

## How It Actually Works — Buffer Pool

```
Disk is slow. RAM is fast. The buffer pool bridges the gap.

        ┌─────────────────────────┐
  SQL → │ Buffer Pool (RAM)       │
        │ ┌─────┐┌─────┐┌─────┐  │
        │ │Page1││Page5││Page9│  │ ← cached pages
        │ └─────┘└─────┘└─────┘  │
        └────────────┬────────────┘
                     │ page fault → read from disk
        ┌────────────┴────────────┐
        │ Disk (data files)        │
        │ Page1 Page2 ... PageN    │
        └─────────────────────────┘

Replacement policy: LRU, Clock, or database-specific.
Dirty pages: Modified in RAM, not yet written to disk.
  → Write-Ahead Log ensures durability before flush.
```

## Why It Matters

- File organization choice directly impacts query performance — heap for writes, sorted for ranges, hash for lookups.
- Understanding the buffer pool explains why "more RAM = faster database" and why repeated queries are fast (cached).
- Real databases combine all methods: heap files + B+ tree indexes + buffer pool.
