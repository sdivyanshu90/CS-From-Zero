## Hook — All or Nothing

You're transferring $500 between bank accounts. The debit goes through. Then the system crashes. Without **transactions**, that $500 is gone forever. Transactions wrap multiple operations into an atomic unit — either everything succeeds or everything is rolled back as if nothing happened.

## Transaction Lifecycle

```sql
BEGIN TRANSACTION;
  UPDATE Account SET balance = balance - 500 WHERE id = 1;
  UPDATE Account SET balance = balance + 500 WHERE id = 2;
COMMIT;  -- both updates permanent

-- If anything fails:
ROLLBACK;  -- undo everything since BEGIN
```

```
States:
  Active → Partially Committed → Committed
    ↓                              
  Failed → Aborted (rolled back)

Once committed: changes survive any failure.
Once aborted: as if the transaction never happened.
```

## See It Work — ACID in Practice

### Atomicity (Write-Ahead Logging)

```
Before modifying data, write the change to a LOG:
  LOG: [TxnID=42, UPDATE Account SET balance=500 WHERE id=1, old=1000]

If crash before COMMIT:
  On recovery, read log → UNDO all changes for uncommitted txns.

If crash after COMMIT:
  On recovery, read log → REDO all changes for committed txns.

ARIES recovery algorithm:
  1. Analysis: Scan log to find active txns at crash
  2. Redo: Replay ALL logged changes (committed or not)
  3. Undo: Reverse changes of uncommitted txns
```

### Isolation Levels

```
Level               Dirty Read  Non-Repeatable  Phantom
─────               ──────────  ──────────────  ───────
READ UNCOMMITTED    Possible    Possible        Possible
READ COMMITTED      No          Possible        Possible
REPEATABLE READ     No          No              Possible
SERIALIZABLE        No          No              No

Dirty Read: See uncommitted data from another txn.
Non-Repeatable Read: Same query, different results (row changed).
Phantom: Same query, different results (rows added/removed).

Default in most DBs: READ COMMITTED.
Strongest: SERIALIZABLE (slowest).
```

### Durability (WAL + Checkpoints)

```
WAL (Write-Ahead Log):
  Changes logged to disk BEFORE data pages are modified.
  Log is append-only → sequential writes → fast!

Checkpoints:
  Periodically flush dirty pages to disk.
  Write checkpoint record to log.
  Recovery only needs to replay log from last checkpoint.
  Without checkpoints: recovery replays entire log (hours).
```

## How It Actually Works — Schedules

```
Serial schedule: Transactions execute one after another.
  T1 complete, then T2 complete. Always correct but slow.

Concurrent schedule: Operations interleaved.
  T1: R(A)  W(A)       R(B) W(B)
  T2:       R(A) W(A)              R(B) W(B)
  
Serializable: A concurrent schedule equivalent to SOME serial order.
  The gold standard — same result as if txns ran sequentially.

Conflict Serializability:
  Two operations conflict if: same data item, different txns,
  at least one is a write.
  Build precedence graph → if acyclic → serializable.
```

## Why It Matters

- Transactions are the foundation of data integrity in every database application.
- Understanding isolation levels prevents bugs: too low → data corruption, too high → poor performance.
- WAL is used by every major database (PostgreSQL, MySQL, SQLite) for crash recovery.
