## Hook — Designing Before Building

You wouldn't build a house without blueprints. Similarly, you don't create a database without a **data model**. The Entity-Relationship (ER) model is the blueprint — a visual way to design your database before writing a single line of SQL.

## Core Concepts

```
Entity: A "thing" with independent existence.
  Student, Course, Department

Attribute: A property of an entity.
  Student: name, id, gpa

Relationship: A connection between entities.
  Student ENROLLS_IN Course

┌──────────┐    enrolls    ┌──────────┐
│ Student   │──────────────│ Course   │
│           │    M : N     │          │
│ id (PK)   │              │ code (PK)│
│ name      │              │ title    │
│ gpa       │              │ credits  │
└──────────┘              └──────────┘
```

## See It Work — Cardinalities

```
One-to-One (1:1):
  Person ──── Passport
  Each person has exactly one passport.

One-to-Many (1:N):
  Department ──── Employee
  One department has many employees.
  Each employee belongs to one department.

Many-to-Many (M:N):
  Student ──── Course
  Students take many courses.
  Courses have many students.
  → Requires a junction/bridge table!

  Student ──── Enrollment ──── Course
               (student_id, course_id, grade)
```

### Keys

```
Primary Key (PK): Uniquely identifies each entity instance.
  Student.id — no two students share an id.

Foreign Key (FK): References a PK in another table.
  Enrollment.student_id → Student.id
  Creates and enforces the relationship.

Candidate Key: Could be PK (unique, not null).
  Student: id, email — both unique, pick one as PK.

Composite Key: PK made of multiple attributes.
  Enrollment: (student_id, course_id) together are unique.
```

## How It Actually Works — ER to Tables

```
ER Diagram:
  Department (1) ──has──→ (N) Employee

Tables:
  CREATE TABLE Department (
    dept_id   INT PRIMARY KEY,
    dept_name VARCHAR(100)
  );

  CREATE TABLE Employee (
    emp_id    INT PRIMARY KEY,
    name      VARCHAR(100),
    dept_id   INT REFERENCES Department(dept_id)  -- FK on "many" side
  );

Rule: FK goes on the "many" side of 1:N relationships.

M:N relationship → create junction table:
  CREATE TABLE Enrollment (
    student_id INT REFERENCES Student(id),
    course_id  INT REFERENCES Course(code),
    grade      CHAR(2),
    PRIMARY KEY (student_id, course_id)
  );
```

### Weak Entities

```
A weak entity can't exist without its owner entity.
  Room can't exist without Building.
  Room number is only unique within a building.

Building (1) ═══has═══ (N) Room
  Room PK: (building_id, room_number)  -- partial key
```

## Why It Matters

- ER modeling prevents costly redesigns later — get the schema right before coding.
- Understanding cardinalities determines where foreign keys go.
- Every database textbook, interview, and real project starts with an ER diagram.
