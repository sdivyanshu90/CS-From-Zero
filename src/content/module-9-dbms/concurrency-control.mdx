## Hook — A Thousand Users, One Database

An airline booking system handles thousands of concurrent requests. Two users try to book the last seat at the same time. Without **concurrency control**, both succeed — but there's only one seat. The database must ensure correctness while maximizing throughput.

## Lock-Based Protocols

```
Shared Lock (S): Multiple txns can read simultaneously.
Exclusive Lock (X): Only one txn can write. Blocks everyone else.

Compatibility:
         Requesting
         S     X
Held S   ✓     ✗
     X   ✗     ✗

Two-Phase Locking (2PL):
  Phase 1 (Growing): Acquire locks, never release.
  Phase 2 (Shrinking): Release locks, never acquire.
  
  Guarantees serializability!
  But: can cause deadlocks (detect with wait-for graph).
```

### Lock Granularity

```
Level          Concurrency   Overhead
─────          ───────────   ────────
Database       Lowest        Minimal
Table          Low           Low
Page           Medium        Medium
Row            High          Higher
Field          Highest       Highest

Most databases: Row-level locking (best balance).
Intention locks: Indicate locking intent at higher levels
to avoid checking every row.
```

## See It Work — MVCC

```
Multi-Version Concurrency Control:
  Keep old versions of data. Readers see a snapshot.
  Writers create new versions. No read-write blocking!

Transaction T1 (started at time 10):
  Sees data as of time 10 (snapshot).

Transaction T2 (started at time 15):
  Updates row X → creates new version with timestamp 15.
  
T1 still reads old version of X (timestamp < 10).
T2 reads new version.

No locks for reads! Reads never block writes!
Used by: PostgreSQL, Oracle, MySQL InnoDB.

PostgreSQL implementation:
  Each row has xmin (creating txn) and xmax (deleting txn).
  Visibility check: xmin committed AND (xmax not set OR xmax not committed).
  Old versions cleaned up by VACUUM.
```

## How It Actually Works — Deadlock Handling

```
T1: LOCK(A), wants LOCK(B)
T2: LOCK(B), wants LOCK(A)
→ Deadlock!

Detection: Build wait-for graph periodically.
  If cycle exists → deadlock.
  
Resolution: Abort one transaction (the "victim").
  Choose victim by: youngest txn, least work done, fewest locks.
  Victim is rolled back, others proceed.

Prevention (alternative):
  Wait-Die: Older txns wait, younger txns abort and retry.
  Wound-Wait: Older txns force younger to abort.
  Timeout: If waiting too long, assume deadlock and abort.
```

## Why It Matters

- Concurrency control enables thousands of users to share a database without corruption.
- MVCC is the dominant approach in modern databases — understanding it explains PostgreSQL and MySQL behavior.
- Deadlock handling is essential for any multi-user application.
