## Hook — Finding a Needle in a Billion Rows

A table has 100 million rows. Without an index, finding one row means scanning all 100 million — minutes of I/O. With a B+ tree index, the same lookup takes 3-4 disk reads — milliseconds. **Indexing** is the single most impactful database optimization technique.

## Why B+ Trees?

```
Binary Search Tree: O(log₂ n) levels.
  100M rows → ~27 levels → 27 disk reads. Still slow!

B+ Tree: Each node holds hundreds of keys (one disk page).
  Branching factor ~500.
  100M rows → log₅₀₀(100M) ≈ 3-4 levels → 3-4 disk reads!

Key insight: Minimize disk I/O, not comparisons.
Each node = one disk page (4-16 KB).
```

## See It Work — B+ Tree Structure

```
Internal nodes: Keys + pointers to children.
Leaf nodes: Keys + pointers to actual data rows.
All leaves linked in a doubly-linked list (range queries!).

                    [50 | 100]
                   /     |     \
          [10|30|45]  [60|75|90]  [110|130]
          /  |  |  \   |  |  |    |   |   \
Leaves: [5,10][12,30][31,45][50,60][61,75][76,90][100,110][111,130]
         ←→    ←→     ←→    ←→     ←→     ←→      ←→       ←→

Range query: WHERE age BETWEEN 30 AND 75
  → Find leaf with 30, then follow linked list to 75.
  → Only reads the needed leaves, not the whole tree.
```

## How It Actually Works — Index Types

```
Clustered Index:
  Table rows physically sorted by the index key.
  Only ONE per table (data can only be sorted one way).
  Range queries are blazing fast (sequential I/O).
  
  PRIMARY KEY creates a clustered index by default.

Non-Clustered (Secondary) Index:
  Separate structure pointing to row locations.
  Multiple per table (index on name, email, etc.).
  Lookup: index → row pointer → fetch row (extra I/O).

Covering Index:
  Index contains ALL columns needed by the query.
  No need to fetch the actual row — answer from index alone!
  CREATE INDEX idx ON Orders(customer_id) INCLUDE (total, date);
```

### When Indexes Hurt

```
Cost of indexes:
  - Extra disk space (index can be 10-30% of table size)
  - Slower writes (every INSERT/UPDATE must update all indexes)
  - Maintenance overhead

Don't index:
  - Small tables (full scan is faster than index lookup)
  - Columns with low selectivity (gender: M/F — half the table)
  - Columns rarely used in WHERE/JOIN/ORDER BY
  - Tables with heavy write workloads
```

### Hash Index vs B+ Tree

```
Hash Index:
  O(1) exact lookup: WHERE id = 42
  Cannot do: ranges, sorting, LIKE, partial matches
  Used: memory-only indexes, hash joins

B+ Tree:
  O(log n) but supports everything:
  =, <, >, <=, >=, BETWEEN, ORDER BY, LIKE 'prefix%'
  Default index type in most databases.
```

## Why It Matters

- Missing indexes are the #1 cause of slow queries in production databases.
- Understanding B+ trees explains why some queries are fast and others aren't.
- Index design is a core DBA skill — adding the right index can make a query 10,000× faster.
