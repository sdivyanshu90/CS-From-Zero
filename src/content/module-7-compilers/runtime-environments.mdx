## Hook — Where Does Your Code Actually Live?

When you call a function, where do local variables go? When you `malloc`, where does memory come from? The **runtime environment** manages memory layout, function calls, parameter passing, and garbage collection. It's the bridge between your compiled code and the operating system.

## Memory Layout of a Process

```
High addresses
┌─────────────────────┐
│       Stack         │ ← grows DOWN
│   (local vars,      │
│    return addrs)     │
├─────────────────────┤
│         ↓           │
│    (free space)      │
│         ↑           │
├─────────────────────┤
│       Heap          │ ← grows UP
│   (malloc/new)       │
├─────────────────────┤
│    BSS (uninitialized│
│     global data)     │
├─────────────────────┤
│    Data (initialized │
│     globals)         │
├─────────────────────┤
│    Text (code)       │ ← read-only
└─────────────────────┘
Low addresses
```

## See It Work — Activation Records (Stack Frames)

Every function call creates an **activation record** on the stack:

```
int factorial(int n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);
}

factorial(3):
┌──────────────────┐
│ factorial(1)     │ ← top of stack
│  n = 1           │
│  return addr     │
├──────────────────┤
│ factorial(2)     │
│  n = 2           │
│  return addr     │
├──────────────────┤
│ factorial(3)     │
│  n = 3           │
│  return addr     │
├──────────────────┤
│ main()           │ ← bottom
└──────────────────┘
```

### What's in a Stack Frame?

```
┌──────────────────────┐ ← previous frame pointer
│ Return address        │
│ Saved frame pointer   │
│ Local variables       │
│ Temporary values      │
│ Saved registers       │
│ Arguments for callee  │
└──────────────────────┘ ← stack pointer
```

## How It Actually Works — Parameter Passing

```
Call by Value:     Copy the value. Changes don't affect caller.
                   C: void f(int x) — x is a copy

Call by Reference:  Pass the address. Changes affect caller.
                   C: void f(int *x) — *x modifies original
                   C++: void f(int &x) — syntactic sugar for pointer

Call by Name:      Re-evaluate the argument expression each time.
                   Historical (Algol 60). Rarely used today.
```

### Calling Conventions

```
x86-64 System V (Linux/macOS):
  First 6 integer args: RDI, RSI, RDX, RCX, R8, R9
  First 8 float args: XMM0-XMM7
  Return value: RAX
  Callee-saved: RBX, RBP, R12-R15

Windows x64:
  First 4 args: RCX, RDX, R8, R9
  Different from Unix! (ABI incompatibility)
```

### Heap Management and Garbage Collection

```
Manual (C/C++):     malloc/free, new/delete
                    Fast but error-prone (leaks, double-free, use-after-free)

Reference Counting: Count pointers to each object. Free when 0.
                    Swift, Python (primary), Rust (Rc/Arc)
                    Problem: circular references!

Mark-and-Sweep:     Trace from roots, mark reachable, sweep unmarked.
                    Java, Go, JavaScript
                    Pause times (stop-the-world)

Generational:       Most objects die young. Check young objects often.
                    Java (G1GC), .NET, V8
                    Optimized for typical allocation patterns.
```

## Why It Matters

- Stack overflows, segfaults, and memory leaks all stem from runtime environment issues.
- Understanding the stack helps you debug recursive functions and security vulnerabilities (buffer overflows).
- Garbage collection tradeoffs explain why some languages have pauses and others don't.
