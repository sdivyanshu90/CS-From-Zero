## Hook — Understanding Structure

The lexer gives us tokens. But `x + y * z` could mean `(x+y)*z` or `x+(y*z)`. The **parser** resolves this ambiguity by building a **parse tree** (or **abstract syntax tree**) according to the grammar rules. Parsing is the heart of the compiler's front end.

## Top-Down Parsing — LL Parsers

Start from the start symbol, try to derive the input string. **Predictive parsing** uses the next token to decide which production to apply.

### Recursive Descent (Hand-Written)

```c
// Grammar: E → T (('+' | '-') T)*
//          T → F (('*' | '/') F)*
//          F → '(' E ')' | NUM | ID

ASTNode* parse_expr() {
    ASTNode *left = parse_term();
    while (token == PLUS || token == MINUS) {
        char op = token;
        advance();
        ASTNode *right = parse_term();
        left = make_binop(op, left, right);
    }
    return left;
}

ASTNode* parse_term() {
    ASTNode *left = parse_factor();
    while (token == STAR || token == SLASH) {
        char op = token;
        advance();
        ASTNode *right = parse_factor();
        left = make_binop(op, left, right);
    }
    return left;
}
```

Recursive descent naturally encodes **operator precedence**: `parse_expr` calls `parse_term` calls `parse_factor` — deeper = tighter binding.

### LL(1) Parsing

```
LL(1): Left-to-right scan, Leftmost derivation, 1 token lookahead

FIRST sets: Which tokens can a production start with?
FOLLOW sets: Which tokens can follow a non-terminal?

Parse table: [non-terminal][token] → production to use
If any cell has multiple entries → NOT LL(1), grammar needs refactoring.
```

## Bottom-Up Parsing — LR Parsers

Start from tokens, reduce to the start symbol. More powerful than LL.

```
Input: id + id * id

Stack          Input              Action
─────          ─────              ──────
               id + id * id $     shift id
id             + id * id $        reduce F → id
F              + id * id $        reduce T → F
T              + id * id $        reduce E → T
E              + id * id $        shift +
E +            id * id $          shift id
E + id         * id $             reduce F → id
E + F          * id $             reduce T → F
E + T          * id $             shift * (not reduce! precedence)
E + T *        id $               shift id
E + T * id     $                  reduce F → id
E + T * F      $                  reduce T → T * F
E + T          $                  reduce E → E + T
E              $                  ACCEPT ✓
```

### LR Parser Variants

```
Parser    States    Power        Usage
──────    ──────    ─────        ─────
LR(0)     Small     Weak         Theoretical
SLR(1)    Small     Medium       Simple cases
LALR(1)   Medium    Strong       Yacc/Bison (most common!)
LR(1)     Large     Strongest    Theory, rarely used directly
```

## How It Actually Works — AST vs Parse Tree

```
Parse Tree (concrete):        AST (abstract):
       E                          +
      /|\                        / \
     E + T                     id   *
     |  /|\                        / \
     T T * F                     id   id
     |  |   |
     F  F  id
     |  |
    id  id

AST drops unnecessary tokens (parentheses, intermediate nodes).
Compilers work with ASTs, not parse trees.
```

## Why It Matters

- Every programming language needs a parser. Understanding parsing helps you design languages and debug syntax errors.
- Recursive descent is used by GCC, Clang, Go, TypeScript, and V8.
- LR parsers (via Yacc/Bison) are used in Ruby (CRuby), PHP, and PostgreSQL's SQL parser.
