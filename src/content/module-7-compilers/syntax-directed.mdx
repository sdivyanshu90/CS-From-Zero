## Hook — Attaching Meaning to Structure

A parse tree tells us the structure of `3 + 4 * 5`. But what does it **mean**? Is the answer 23 or 35? **Syntax-Directed Translation (SDT)** attaches **semantic rules** to grammar productions, computing values, types, or code as the parser builds the tree.

## Syntax-Directed Definitions (SDD)

An SDD associates **attributes** with grammar symbols and **semantic rules** with productions:

```
Production          Semantic Rule
──────────          ─────────────
E → E₁ + T          E.val = E₁.val + T.val
E → T               E.val = T.val
T → T₁ * F          T.val = T₁.val * F.val
T → F               T.val = F.val
F → ( E )           F.val = E.val
F → num              F.val = num.lexval

Computing "3 + 4 * 5":
  F.val = 5 → T.val = 4 * 5 = 20 → E.val = 3 + 20 = 23
```

### Synthesized vs. Inherited Attributes

```
Synthesized: Value flows UP the tree (child → parent)
  Example: E.val computed from children's .val
  Can be evaluated bottom-up (during LR parsing)

Inherited: Value flows DOWN the tree (parent → child)
  Example: Type information passed from declaration to uses
  Requires multiple passes or special evaluation order
```

## See It Work — Type Checking

```
Declaration: int x;
Expression:  x + 3.14

SDD for type checking:
  E → E₁ + E₂
    E.type = if (E₁.type == E₂.type) then E₁.type
             else if compatible(E₁.type, E₂.type)
             then coerce(E₁.type, E₂.type)  // int → float
             else TYPE_ERROR

  x + 3.14: int + float → coerce to float → result is float
```

### Symbol Tables

```
int x = 5;        → symbol_table["x"] = {type: int, offset: 0}
float y = 3.14;   → symbol_table["y"] = {type: float, offset: 4}
x + y;             → lookup both, check type compatibility

Scoping: nested symbol tables (stack of hash tables)
  {                         ← push new scope
    int x = 10;             ← shadows outer x
    {                       ← push another scope
      float x = 3.14;      ← shadows again
    }                       ← pop scope
  }                         ← pop scope
```

## How It Actually Works — Translation Schemes

**Translation schemes** embed semantic actions directly in the grammar:

```
E → E₁ + T   { emit("ADD"); }
E → T
T → T₁ * F   { emit("MUL"); }
T → F
F → num       { emit("PUSH " + num.val); }

Parsing "3 + 4 * 5":
  PUSH 3
  PUSH 4
  PUSH 5
  MUL
  ADD

This generates stack-machine code during parsing!
```

## Why It Matters

- SDT is how compilers give meaning to syntax — type checking, constant folding, and code generation all use it.
- Symbol tables track every variable, function, and type in your program.
- Understanding SDT explains why certain code patterns produce type errors.
