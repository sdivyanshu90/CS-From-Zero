## Hook — Reliable vs Fast

IP delivers packets "best effort" — packets can be lost, duplicated, or reordered. **TCP** adds reliability: guaranteed delivery, correct order, flow control. **UDP** says "just send it fast." Choosing between them is one of the most important decisions in network programming.

## UDP — User Datagram Protocol

```
UDP Header (8 bytes):
┌──────────┬──────────┐
│ Src Port │ Dst Port │  4 bytes
├──────────┼──────────┤
│ Length   │ Checksum │  4 bytes
└──────────┴──────────┘

Features:
  ✗ No connection setup
  ✗ No guaranteed delivery
  ✗ No ordering
  ✗ No flow/congestion control
  ✓ Fast (minimal overhead)
  ✓ Low latency
  ✓ Multicast support

Used for: DNS, video streaming, VoIP, gaming, NTP
  → Where speed matters more than reliability
  → Lost frame in video? Show next one. Don't wait.
```

## TCP — Transmission Control Protocol

```
TCP Header (20+ bytes):
┌──────────┬──────────┐
│ Src Port │ Dst Port │
├──────────┴──────────┤
│ Sequence Number      │
├──────────────────────┤
│ Acknowledgment Number│
├──────┬───────────────┤
│Flags │ Window Size   │  SYN, ACK, FIN, RST, PSH
├──────┴───────────────┤
│ Checksum │ Urg Ptr   │
└──────────────────────┘
```

### Three-Way Handshake

```
Client              Server
  │── SYN (seq=100) ──→│   "I want to connect"
  │←─ SYN-ACK ────────│   "OK, here's my seq too"
  │   (seq=300,ack=101)│
  │── ACK (ack=301) ──→│   "Got it, connection open"
  
Connection established. Both sides know initial sequence numbers.

Closing: FIN → ACK, FIN → ACK (four-way)
```

### Reliable Delivery

```
Sequence numbers track every byte sent:
  Segment 1: seq=100, 500 bytes → covers bytes 100-599
  Segment 2: seq=600, 500 bytes → covers bytes 600-1099

Receiver ACKs: "I've received up to byte 1100" (cumulative ACK)

Lost segment? Receiver ACKs last received byte.
  3 duplicate ACKs → sender retransmits (Fast Retransmit).
  Timeout → sender retransmits.

Selective ACK (SACK): "I have bytes 100-599 and 1100-1599,
  missing 600-1099." Only retransmit missing segment.
```

### Flow Control

```
Receiver advertises window size in ACK:
  "I have 16KB of buffer space free"

Sender limits unacknowledged data to window size.
  → Prevents fast sender from overwhelming slow receiver.

Window = 0 → Sender stops. Periodically sends probe.
  When receiver has space → window > 0 → resume.
```

## Why It Matters

- TCP vs UDP is a core networking decision that affects every networked application.
- Understanding TCP explains why web pages load reliably and why video calls sometimes stutter.
- Flow control and congestion control prevent the internet from collapsing under load.
