## Hook — Preventing Internet Meltdown

In 1986, the internet nearly collapsed. Senders kept retransmitting into an already overloaded network, making congestion worse — a positive feedback loop called **congestion collapse**. Van Jacobson's congestion control algorithms saved the internet and remain in use today.

## Flow Control vs Congestion Control

```
Flow Control:
  Prevent overwhelming the RECEIVER.
  Receiver advertises window size.
  Sender respects it.
  End-to-end (sender ↔ receiver).

Congestion Control:
  Prevent overwhelming the NETWORK.
  Sender infers congestion from packet loss/delay.
  No explicit signal from network (usually).
  End-to-end (sender guesses network state).

Effective window = min(receiver window, congestion window)
```

## See It Work — TCP Congestion Control

### Slow Start

```
Start with cwnd (congestion window) = 1 MSS.
For each ACK received: cwnd += 1 MSS.
→ Window DOUBLES every RTT (exponential growth).

RTT 1: cwnd = 1  → send 1 segment
RTT 2: cwnd = 2  → send 2 segments
RTT 3: cwnd = 4  → send 4 segments
RTT 4: cwnd = 8  → send 8 segments

Grows until:
  - Packet loss → cut cwnd
  - Reaches ssthresh → switch to congestion avoidance
```

### Congestion Avoidance

```
After reaching ssthresh:
  For each ACK: cwnd += 1/cwnd (additive increase)
  → Window grows by ~1 MSS per RTT (linear growth).

On packet loss (timeout):
  ssthresh = cwnd / 2
  cwnd = 1 MSS
  → Back to slow start!

AIMD (Additive Increase, Multiplicative Decrease):
  Increase: +1 per RTT (linear, gentle probe)
  Decrease: ÷2 on loss (aggressive cut)
  
  Creates a "sawtooth" pattern:
  cwnd  ╱╲  ╱╲  ╱╲
       ╱  ╲╱  ╲╱  ╲
       time →
```

### Fast Recovery (TCP Reno)

```
3 duplicate ACKs (mild congestion, not timeout):
  ssthresh = cwnd / 2
  cwnd = ssthresh (NOT back to 1!)
  → Skip slow start, resume congestion avoidance.

Much faster recovery than restarting from cwnd=1.
```

## How It Actually Works — Modern Algorithms

```
Algorithm    Approach              Used By
─────────    ────────              ───────
Reno         Loss-based (AIMD)     Classic, still common
Cubic        Loss-based (cubic fn) Linux default
BBR          Model-based           Google (YouTube, Cloud)
             (bandwidth + RTT)

Cubic: Grows faster far from last loss, slower near it.
  Cubic function of time since last loss event.
  Better than Reno for high-bandwidth, high-latency links.

BBR (Bottleneck Bandwidth and RTT):
  Instead of waiting for loss, BBR:
  1. Estimates bottleneck bandwidth (max delivery rate)
  2. Estimates minimum RTT
  3. Targets: send at bandwidth × min_RTT
  
  Better on links with large buffers (doesn't need loss signal).
  Used by Google for YouTube — 4-14% throughput improvement.
```

## Why It Matters

- Without congestion control, the internet would collapse under heavy load.
- Understanding AIMD explains why downloads start slow and speed up.
- BBR represents a paradigm shift — measuring bandwidth instead of reacting to loss.
