## Hook

Think about scientific notation in everyday life. Instead of writing 299,792,458 (the speed of light in m/s), scientists write 2.998 × 10⁸. They store a small significant number and then shift the decimal point by an exponent. This lets them handle both astronomically large numbers and microscopically small ones with the same compact format.

In computing, this is called **floating point representation** — a way to store fractions and very large/small numbers by recording a significand and an exponent, much like scientific notation.

## What Is It?

Computers store numbers as <ConceptLink to="digital-logic/what-is-information" label="bits" />. For whole numbers, <ConceptLink to="digital-logic/number-systems" label="two's complement" /> works perfectly. But for fractions (like 3.14) or huge numbers (like 6.02 × 10²³), we need a different encoding.

There are two approaches:

- **Fixed point**: The decimal point is always at a fixed position (e.g., 8 bits for the integer part, 8 bits for the fraction). Simple but limited in range.
- **Floating point**: The decimal point "floats" — its position is encoded in the number itself, trading precision for enormous range.

The world standard for floating point is **IEEE 754**, used by virtually every modern processor.

## See It Work

### Fixed Point Example

With 8 bits split 4:4 (4 integer bits, 4 fraction bits):

```
Binary:   0101 . 1100
Integer:  0×8 + 1×4 + 0×2 + 1×1 = 5
Fraction: 1×0.5 + 1×0.25 + 0×0.125 + 0×0.0625 = 0.75

Value: 5.75
```

Simple! But the range is tiny (0 to 15.9375 in this format) and you can't represent very large or very small numbers.

### IEEE 754 Single Precision (32-bit float)

A 32-bit float has three parts:

```
| Sign (1 bit) | Exponent (8 bits) | Mantissa (23 bits) |
|     0/1      |    biased exp     |   fractional part   |
```

- **Sign**: 0 = positive, 1 = negative.
- **Exponent**: Stored with a bias of 127. The actual exponent = stored value − 127.
- **Mantissa** (significand): The fractional part after an implied leading 1. (The "1." is free — you don't store it!)

### Encoding Example: −6.5

```
Step 1: Convert to binary
  6 = 110, 0.5 = .1
  6.5 = 110.1

Step 2: Normalize (scientific notation, base 2)
  110.1 = 1.101 × 2²

Step 3: Extract the parts
  Sign = 1 (negative)
  Exponent = 2 + 127 (bias) = 129 = 10000001
  Mantissa = 101 (drop the leading "1.")
             pad to 23 bits: 10100000000000000000000

Result: 1 10000001 10100000000000000000000
```

## How It Actually Works

### Special Values

IEEE 754 reserves certain bit patterns for special cases:

| Exponent (all) | Mantissa (all) | Meaning |
|----------------|----------------|---------|
| 0 | 0 | Zero (+0 or −0) |
| 0 | nonzero | Denormalized (very tiny numbers near zero) |
| 255 (all 1s) | 0 | Infinity (+∞ or −∞) |
| 255 (all 1s) | nonzero | NaN (Not a Number) |

`NaN` appears when you do things like divide 0 by 0 or take the square root of a negative number.

### Precision Limitations

Floating point cannot represent all decimal fractions exactly. The classic example:

```
0.1 + 0.2 = 0.30000000000000004   (not 0.3!)
```

Why? Because 0.1 in binary is a repeating fraction (like 1/3 = 0.333... in decimal). With only 23 bits of mantissa, the representation is *approximate*.

This matters in:
- **Financial software**: Never use floating point for money! Use integers (cents) instead.
- **Scientific computing**: Errors accumulate over millions of operations.
- **Game physics**: Small precision errors can cause objects to jitter or drift.

### Double Precision (64-bit)

For more precision, use 64-bit doubles: 1 sign bit, 11 exponent bits, 52 mantissa bits. This gives about 15–17 significant decimal digits instead of 7.

## Why It Matters

Floating point is everywhere:

- **Graphics**: Every 3D game calculates positions, lighting, and physics with floating point.
- **Machine learning**: Neural networks multiply billions of floating-point weights.
- **Science**: Simulations of weather, protein folding, and galaxies rely on double-precision floats.
- **The Ariane 5 explosion (1996)**: A 64-bit float was converted to a 16-bit integer. The number was too large, causing an overflow. The rocket self-destructed 37 seconds after launch. Understanding floating point prevents real-world disasters.
