## Hook — The Kernel's API

Every time your program opens a file, allocates memory, or sends a network packet, it makes a **system call** — a controlled request to the kernel. System calls are the boundary between your code and the operating system, and understanding them is understanding how software truly interacts with hardware.

## What Is a System Call?

```
User program:          Kernel:
printf("hello")
  → write(1, "hello", 5)
    → syscall instruction     → validate args
      (switch to ring 0)      → find fd 1 (stdout)
                               → write to terminal driver
      ← return to ring 3     ← return bytes written (5)
```

### The System Call Journey

```
1. Program calls wrapper function: write(fd, buf, count)
2. Wrapper puts syscall number in register (e.g., RAX = 1 for write)
3. Arguments in registers: RDI=fd, RSI=buf, RDX=count
4. Execute 'syscall' instruction → CPU switches to kernel mode
5. Kernel looks up handler in syscall table[RAX]
6. Handler validates args, performs operation
7. Returns result in RAX (or negative errno on error)
8. 'sysret' instruction → back to user mode
```

## See It Work — Essential System Calls

```
Category        Calls                    Purpose
────────        ─────                    ───────
Process         fork, exec, wait, exit   Create, run, manage processes
File            open, read, write, close File I/O
                lseek, stat, chmod
Directory       mkdir, rmdir, readdir    Directory operations
Memory          mmap, munmap, brk        Memory management
Network         socket, bind, listen     Network communication
                accept, connect,
                send, recv
Signal          signal, kill, sigaction   Asynchronous notification
IPC             pipe, shmget, msgget     Inter-process communication
```

### strace — Watching System Calls

```bash
$ strace ls
execve("/bin/ls", ["ls"], ...) = 0
openat(AT_FDCWD, ".", O_RDONLY) = 3
getdents64(3, ...) = 480
write(1, "file1.txt\nfile2.txt\n", 20) = 20
close(3) = 0
exit_group(0) = ?

# Every interaction with the OS is visible!
# Incredibly useful for debugging
```

## How It Actually Works — User vs Kernel

```
Library function (user space):     System call (kernel space):
─────────────────────────────      ──────────────────────────
printf() → formats string         write() → kernel validates,
  → calls write()                   writes to device
malloc() → manages heap           brk()/mmap() → kernel maps
  → calls brk() or mmap()          physical pages
fopen() → creates FILE struct     open() → kernel creates fd,
  → calls open()                    checks permissions

C library (glibc/musl) wraps syscalls with buffering,
error handling, and convenience features.
```

### Why Not Just Call the Kernel Directly?

```
1. Portability: Library functions are standardized (POSIX).
   Syscall numbers/conventions differ between Linux, macOS, BSDs.

2. Buffering: printf/fwrite buffer in user space.
   Calling write() for every character = massive overhead.

3. Abstraction: malloc manages a heap with free lists.
   Calling mmap for every allocation = wasteful (min 4KB page).
```

## Why It Matters

- System calls are the ONLY way user programs interact with the OS kernel.
- Every performance-sensitive application must minimize syscalls (batch I/O, buffer data).
- strace is one of the most powerful debugging tools — it shows exactly what your program asks the OS to do.
