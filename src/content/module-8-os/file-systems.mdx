## Hook — Persistent Storage That Makes Sense

A hard drive is just a sequence of bytes. Yet you see folders, filenames, and permissions. The **file system** creates this abstraction — organizing raw storage into a hierarchical, named, protected structure. Understanding file systems explains how data persists and why disk performance matters.

## File System Structure

```
Application:     open("file.txt")
                      ↓
Logical FS:      Lookup name → inode number
                      ↓
File Organization: inode → data block locations
                      ↓
Basic FS:        Read block 4523 from disk
                      ↓
I/O Control:     Send command to disk controller
                      ↓
Device:          Physical read from platter/flash
```

## See It Work — Unix File System (ext4)

### Inodes

```
Every file has an inode (index node):
┌──────────────────────────┐
│ Inode #12345              │
│ Type: regular file        │
│ Permissions: rwxr-xr-x   │
│ Owner: somu               │
│ Size: 8192 bytes          │
│ Timestamps: atime, mtime  │
│ Link count: 1             │
│ Direct blocks: [100, 101] │ ← first 12 blocks (48KB)
│ Indirect: [500]           │ ← points to block of pointers
│ Double indirect: [600]    │ ← pointer to pointers to pointers
│ Triple indirect: [700]    │ ← for huge files
└──────────────────────────┘

File data is in blocks (typically 4KB each).
Filename is NOT in the inode — it's in the directory!
```

### Directories

```
A directory is just a file containing (name, inode) pairs:

Directory /home/somu/:
┌──────────────────────┐
│ "."        → inode 50 │  ← current directory
│ ".."       → inode 2  │  ← parent directory
│ "file.txt" → inode 12345 │
│ "code/"    → inode 67890 │
└──────────────────────┘

Hard link: Multiple names pointing to same inode.
Symbolic link: A file containing a path string.
```

### Journaling (Crash Recovery)

```
Problem: Power loss during write → inconsistent filesystem.
  Writing file requires updating: data block, inode, bitmap, directory.
  If only some updates complete → corruption!

Solution: Write-Ahead Logging (Journal)
  1. Write ALL changes to journal first
  2. Then apply changes to actual locations
  3. If crash during step 2: replay journal on reboot

ext4 journal modes:
  journal:  Log data + metadata (safest, slowest)
  ordered:  Log metadata, write data first (default, good balance)
  writeback: Log metadata only (fastest, risk of stale data)
```

## How It Actually Works — Modern File Systems

```
File System    OS         Features
───────────    ──        ────────
ext4           Linux      Journaling, extents, default Linux FS
XFS            Linux      High performance, large files
Btrfs          Linux      Copy-on-write, snapshots, checksums
ZFS            FreeBSD    Enterprise, RAID, deduplication, checksums
NTFS           Windows    Journaling, ACLs, compression
APFS           macOS/iOS  Copy-on-write, encryption, snapshots
FAT32          Universal  Simple, no journaling, 4GB file limit
```

## Why It Matters

- File systems determine data reliability, performance, and recoverability after crashes.
- Understanding inodes explains hard links, `ls -i`, and why "out of space" can happen with free blocks (out of inodes!).
- Journaling vs. non-journaling is the difference between "fsck for 2 hours" and "reboot in 5 seconds."
