## Hook — When Threads Collide

Two threads increment a shared counter: both read 5, both write 6. The counter should be 7, but it's 6. This **race condition** is the fundamental problem of concurrent programming. **Synchronization** primitives — mutexes, semaphores, condition variables — ensure threads cooperate correctly.

## The Critical Section Problem

```
Thread A:              Thread B:
  read counter (5)       read counter (5)
  add 1                  add 1
  write counter (6)      write counter (6)

Expected: 7    Actual: 6 — LOST UPDATE!

The increment isn't atomic: read-modify-write is 3 steps.
A context switch between any steps causes a race condition.
```

### Requirements for a Correct Solution

```
1. Mutual Exclusion: Only one thread in the critical section at a time.
2. Progress: If no one is in the CS, a waiting thread can enter.
3. Bounded Waiting: No thread waits forever (no starvation).
```

## See It Work — Synchronization Primitives

### Mutex (Mutual Exclusion Lock)

```c
pthread_mutex_t lock;

void increment() {
    pthread_mutex_lock(&lock);    // ACQUIRE — blocks if locked
    counter++;                     // Critical section — safe!
    pthread_mutex_unlock(&lock);  // RELEASE — wake waiting threads
}
```

### Semaphore

```c
sem_t sem;
sem_init(&sem, 0, 3);  // Allow up to 3 concurrent accesses

void access_resource() {
    sem_wait(&sem);     // Decrement. If 0, block.
    use_resource();     // Up to 3 threads can be here
    sem_post(&sem);     // Increment. Wake a waiter.
}

// Mutex = semaphore with initial value 1 (binary semaphore)
```

### Condition Variable

```c
pthread_mutex_t lock;
pthread_cond_t not_empty;
int items = 0;

// Producer
pthread_mutex_lock(&lock);
items++;
pthread_cond_signal(&not_empty);  // Wake one waiter
pthread_mutex_unlock(&lock);

// Consumer
pthread_mutex_lock(&lock);
while (items == 0)                // MUST use while, not if!
    pthread_cond_wait(&not_empty, &lock);  // Release lock + sleep
items--;
pthread_mutex_unlock(&lock);
```

## How It Actually Works — Classic Problems

### Producer-Consumer (Bounded Buffer)

```
Producer → [buffer of size N] → Consumer

Semaphores: empty=N, full=0, mutex=1

Producer:                    Consumer:
  wait(empty)                  wait(full)
  wait(mutex)                  wait(mutex)
  add item to buffer           remove item from buffer
  signal(mutex)                signal(mutex)
  signal(full)                 signal(empty)
```

### Readers-Writers

```
Multiple readers can read simultaneously.
Writers need exclusive access.

Reader:                    Writer:
  wait(reader_lock)          wait(write_lock)
  read_count++               write data
  if (read_count == 1)       signal(write_lock)
    wait(write_lock)
  signal(reader_lock)
  ... read ...
  wait(reader_lock)
  read_count--
  if (read_count == 0)
    signal(write_lock)
  signal(reader_lock)
```

## Why It Matters

- Race conditions cause some of the hardest bugs: intermittent, non-reproducible, data-corrupting.
- Every multi-threaded server, database, and operating system uses these primitives.
- Understanding synchronization is essential for writing correct concurrent code.
