## Hook — The Bottleneck

RAM access: ~100 nanoseconds. SSD: ~100 microseconds. HDD: ~10 milliseconds. That's a 100,000× gap between RAM and HDD. **I/O scheduling** minimizes this bottleneck by reordering disk requests to reduce mechanical seek time and maximize throughput.

## How Disks Work

```
HDD (Hard Disk Drive):
  Spinning platter, moving read/write head.
  Seek time: Move head to correct track (~5-10 ms)
  Rotational delay: Wait for sector (~4 ms at 7200 RPM)
  Transfer: Read data (~0.1 ms per 4KB)
  
  Total: ~10 ms per random read. Sequential: much faster!

SSD (Solid State Drive):
  No moving parts. Flash memory chips.
  Random read: ~100 μs (100× faster than HDD)
  Sequential: ~500 MB/s (similar to HDD)
  No seek time → scheduling less critical
  But: write amplification, wear leveling needed
```

## See It Work — Disk Scheduling Algorithms

### FCFS (First Come, First Served)

```
Head at track 50. Queue: 98, 183, 37, 122, 14, 124, 65, 67

Movement: 50→98→183→37→122→14→124→65→67
Total head movement: 640 tracks

Simple but terrible — lots of unnecessary seeking.
```

### SSTF (Shortest Seek Time First)

```
Head at 50. Always go to nearest request.
50→37→14→65→67→98→122→124→183
Total: 236 tracks — much better!

Problem: Starvation of far-away requests.
```

### SCAN (Elevator Algorithm)

```
Head at 50, moving UP. Go to end, then reverse.
50→65→67→98→122→124→183→(end)→37→14
Total: ~332 tracks

Fair: no starvation. Predictable latency.
Like an elevator — services requests in one direction,
then reverses.
```

### C-SCAN (Circular SCAN)

```
Only service requests in one direction.
At the end, jump back to beginning without servicing.
50→65→67→98→122→124→183→(jump to 0)→14→37
More uniform wait times than SCAN.
```

## How It Actually Works — Modern I/O

```
Linux I/O schedulers:
  mq-deadline: Deadline-based, prevents starvation (default for HDD)
  bfq: Budget Fair Queuing, good for interactive desktops
  none/noop: No reordering (best for SSDs — no seek time!)

I/O stack:
  Application → VFS → File System → Block Layer → I/O Scheduler
  → Device Driver → Hardware
```

### Buffered vs. Direct I/O

```
Buffered I/O (default):
  Write goes to page cache → OS writes to disk later
  Read checks page cache first → disk only on miss
  + Fast for repeated/sequential access
  - Data loss risk if crash before flush

Direct I/O (O_DIRECT):
  Bypass page cache, write directly to disk
  Used by databases (they have their own cache)
  + Predictable performance, no double-caching
  - Slower for general use
```

## Why It Matters

- Disk I/O is the #1 performance bottleneck in most applications.
- Choosing the right I/O scheduler matters for HDDs (less so for SSDs).
- Understanding buffered vs. direct I/O is critical for database and server performance.
