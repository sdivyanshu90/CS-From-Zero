## Hook ‚Äî The Deadly Embrace

Thread A holds Lock 1 and wants Lock 2. Thread B holds Lock 2 and wants Lock 1. Neither can proceed. Both wait forever. This is **deadlock** ‚Äî one of the most dangerous bugs in concurrent systems. Understanding it is essential for writing reliable multi-threaded code.

## Four Necessary Conditions

All four must hold simultaneously for deadlock to occur:

```
1. Mutual Exclusion: Resources can't be shared
2. Hold and Wait: Holding resources while waiting for more
3. No Preemption: Resources can't be forcibly taken away
4. Circular Wait: A‚ÜíB‚ÜíC‚Üí...‚ÜíA chain of waiting

Break ANY one condition ‚Üí no deadlock possible
```

## See It Work ‚Äî Deadlock Example

```c
// Thread A:                  // Thread B:
pthread_mutex_lock(&lock1);   pthread_mutex_lock(&lock2);
// ... do work ...            // ... do work ...
pthread_mutex_lock(&lock2);   pthread_mutex_lock(&lock1);
// DEADLOCK!                  // DEADLOCK!
// A holds lock1, waits lock2 // B holds lock2, waits lock1
```

### Resource Allocation Graph

```
Thread A ‚Üí Lock 2 (wants)     Lock 1 ‚Üí Thread A (held by)
Thread B ‚Üí Lock 1 (wants)     Lock 2 ‚Üí Thread B (held by)

Cycle detected! ‚Üí Deadlock.

    A ‚îÄ‚îÄwants‚îÄ‚îÄ‚Üí L2
    ‚Üë             ‚îÇ
  held by      held by
    ‚îÇ             ‚Üì
    L1 ‚Üê‚îÄ‚îÄwants‚îÄ‚îÄ B
```

## How It Actually Works ‚Äî Strategies

### Prevention (Break a Condition)

```
Break Circular Wait (most practical):
  Order all locks globally (e.g., by address).
  Always acquire in order: lock1 before lock2.

// Fixed:
// Thread A:                  // Thread B:
pthread_mutex_lock(&lock1);   pthread_mutex_lock(&lock1);  // same order!
pthread_mutex_lock(&lock2);   pthread_mutex_lock(&lock2);
// No deadlock possible ‚Äî both acquire in same order.

Break Hold-and-Wait:
  Acquire ALL needed locks at once (atomic).
  Impractical if you don't know all locks in advance.

Break No-Preemption:
  If you can't get a lock, release what you hold and retry.
  pthread_mutex_trylock() ‚Äî non-blocking attempt.
```

### Avoidance (Banker's Algorithm)

```
Before granting a resource, check if the system stays in a "safe state"
(a state where every process can eventually finish).

Process   Max   Allocated   Need   Available
P1        10    5           5      3
P2        4     2           2
P3        9     2           7

Safe sequence: P2(needs 2, avail 3‚Üífinish‚Üíreturn 2‚Üíavail 5),
               P1(needs 5, avail 5‚Üífinish‚Üíreturn 5‚Üíavail 10),
               P3(needs 7, avail 10‚Üífinish)

If granting a request makes no safe sequence possible ‚Üí deny it.
O(n¬≤m) per request ‚Äî expensive but prevents deadlock.
```

### Detection and Recovery

```
Periodically check for cycles in the wait-for graph.
If deadlock detected:
  - Kill one process (victim selection)
  - Roll back to checkpoint
  - Preempt resources

Most real systems: just detect and kill (databases do this).
```

## Why It Matters

- Deadlocks crash servers, freeze databases, and hang operating systems.
- Lock ordering is the most practical prevention technique ‚Äî used everywhere.
- Understanding deadlock helps you design concurrent systems that don't get stuck.
