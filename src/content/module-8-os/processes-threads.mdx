## Hook вҖ” Programs in Motion

A program sitting on disk is just bytes. When you run it, the OS creates a **process** вҖ” a running instance with its own memory, file handles, and execution state. Modern programs use **threads** to do multiple things simultaneously within the same process. Understanding processes and threads is essential for concurrent programming.

## See It Work вҖ” Fork and Exec (Unix)

```c
#include <unistd.h>
#include <sys/wait.h>

int main() {
    pid_t pid = fork();   // Create child process (copy of parent)
    
    if (pid == 0) {
        // Child process
        execl("/bin/ls", "ls", "-la", NULL);  // Replace with new program
    } else {
        // Parent process
        wait(NULL);  // Wait for child to finish
        printf("Child done\n");
    }
}

// fork() creates a COPY of the process
// exec() REPLACES the process image with a new program
// Together: fork+exec = run a new program
```

## How It Actually Works вҖ” Thread Models

```
User-level threads (green threads):
  Managed by runtime, not OS. Fast creation/switching.
  But: one blocking syscall blocks ALL threads.
  Used by: early Java, Go goroutines (M:N model)

Kernel-level threads:
  Managed by OS. True parallelism on multi-core.
  But: creation/switching more expensive.
  Used by: Linux pthreads, Windows threads

M:N model (hybrid):
  M user threads mapped to N kernel threads.
  Best of both: cheap creation + true parallelism.
  Used by: Go runtime, Erlang BEAM
```

## Why It Matters

- Multi-threaded programming is essential for responsive UIs, servers, and performance.
- Process isolation is why your browser runs tabs in separate processes вҖ” one crash doesn't kill all tabs.
- Understanding threads explains race conditions, deadlocks, and why concurrent programming is hard.
