## Hook — Who Runs Next?

Your computer runs hundreds of processes but has only a few CPU cores. The **scheduler** decides which process runs, for how long, and when to switch. A good scheduler keeps the CPU busy, responds quickly to user input, and treats all processes fairly. It's one of the most critical OS components.

## Scheduling Criteria

```
Metric               Goal              Matters for
──────               ────              ───────────
CPU utilization      Maximize          Server efficiency
Throughput           Maximize          Batch processing
Turnaround time      Minimize          Total completion time
Waiting time         Minimize          Fairness
Response time        Minimize          Interactive use
```

## See It Work — Scheduling Algorithms

### First Come, First Served (FCFS)

```
Process    Burst time
P1         24
P2         3
P3         3

Order: P1 → P2 → P3
Timeline: |---P1(24)---|--P2(3)--|--P3(3)--|

Avg waiting time: (0 + 24 + 27) / 3 = 17

Problem: CONVOY EFFECT — short processes stuck behind long ones
```

### Shortest Job First (SJF)

```
Order: P2 → P3 → P1
Timeline: |P2(3)|P3(3)|---P1(24)---|

Avg waiting time: (6 + 0 + 3) / 3 = 3   ← much better!

Optimal for avg waiting time, but:
- Can't know burst time in advance (must predict)
- Starvation: long processes may never run
```

### Round Robin (RR)

```
Time quantum = 4

Timeline: |P1(4)|P2(3)|P3(3)|P1(4)|P1(4)|P1(4)|P1(4)|P1(4)|

Every process gets at most 4ms, then goes to back of queue.
Fair, good response time, but more context switches.

Quantum tradeoff:
  Too large → becomes FCFS
  Too small → too many context switches (overhead)
  Typical: 10-100 ms
```

### Multi-Level Feedback Queue (MLFQ)

```
Queue 0 (highest priority, RR quantum=8ms):
  New processes start here.
  If they use full quantum → demote to Queue 1.

Queue 1 (medium priority, RR quantum=16ms):
  If they use full quantum → demote to Queue 2.

Queue 2 (lowest priority, FCFS):
  Long-running (CPU-bound) processes end up here.

Key insight: Interactive processes voluntarily give up CPU
(I/O wait) before quantum expires → stay in high-priority queue.
CPU-bound processes use full quantum → get demoted.

This automatically separates interactive from batch processes!
```

## How It Actually Works — Linux CFS

Linux uses the **Completely Fair Scheduler (CFS)**:

```
Idea: Track "virtual runtime" for each process.
      Always run the process with the LEAST virtual runtime.
      Uses a red-black tree for O(log n) scheduling.

vruntime increases proportional to actual runtime / weight.
Higher-priority processes have larger weight → vruntime increases slower.

Result: All processes get fair CPU share proportional to their priority.
No starvation possible — everyone's vruntime eventually becomes smallest.
```

## Why It Matters

- Scheduling directly affects system responsiveness and throughput.
- Understanding scheduling helps diagnose performance issues (is your process being starved?).
- The same concepts apply to task scheduling in distributed systems, databases, and network queues.
