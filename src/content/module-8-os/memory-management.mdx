## Hook — The Illusion of Infinite Memory

Your laptop has 16 GB of RAM, yet each process thinks it has its own 256 TB address space. How? **Virtual memory** creates an abstraction where each process sees a private, contiguous address space mapped to physical RAM by the hardware. This is one of the most elegant abstractions in all of computing.

## Virtual Memory

```
Process A sees:            Physical RAM:
0x0000 ┌──────┐           ┌──────┐ 0x0000
       │ Code │──────────→│ A:Code│
       │ Data │──────┐    │ B:Code│
       │ Heap │─┐   └───→│ A:Data│
       │      │ │         │ B:Heap│
       │ Stack│─┤    ┌───→│ A:Heap│
0xFFFF └──────┘ │    │    │ A:Stk │
                │    │    │ B:Data│
Process B sees: │    │    │ B:Stk │
0x0000 ┌──────┐ │    │    └──────┘ 0xFFFF
       │ Code │─┤    │
       │ Data │─┤    │    Disk (swap):
       │ Heap │─┘    │    ┌──────┐
       │ Stack│──────┘    │unused │
0xFFFF └──────┘           │pages  │
                          └──────┘
Same virtual addresses → different physical locations!
```

## See It Work — Paging

Memory is divided into fixed-size **pages** (typically 4KB):

```
Virtual address: 0x12345678
  Page number: 0x12345 (upper bits)
  Offset: 0x678 (lower 12 bits, within the page)

Page Table (per process):
  Virtual Page → Physical Frame
  0x12345     → 0xABCDE        (in RAM)
  0x12346     → DISK (page fault! load from swap)
  0x12347     → 0x00001        (in RAM)

Translation: Virtual 0x12345678 → Physical 0xABCDE678
```

### Translation Lookaside Buffer (TLB)

```
Page table is in RAM → every memory access needs TWO accesses!
  1. Access page table to get physical address
  2. Access the actual data

TLB: Hardware cache of recent page table entries.
  TLB hit (99%+): translation in 1 cycle
  TLB miss: walk page table (~100 cycles)

TLB is why context switches are expensive:
  Different process → different page table → flush TLB!
```

### Multi-Level Page Tables

```
Problem: 2⁴⁸ byte address space / 4KB pages = 2³⁶ entries.
         Each entry 8 bytes = 512 GB just for the page table!

Solution: Multi-level (x86-64 uses 4 levels):
  Level 4 (PML4): 512 entries → each points to...
  Level 3 (PDPT): 512 entries → each points to...
  Level 2 (PD):   512 entries → each points to...
  Level 1 (PT):   512 entries → physical frame

Only allocate page table pages for addresses actually used!
Most entries at upper levels are empty.
Sparse usage = small page table.
```

## How It Actually Works — Page Replacement

When RAM is full and a page fault occurs, which page to evict?

```
Algorithm       Description                  Performance
─────────       ───────────                  ───────────
FIFO            Evict oldest page            Poor (Belady's anomaly)
LRU             Evict least recently used    Good but expensive
Clock           Approximate LRU             Good, practical
OPT             Evict page used farthest     Optimal (impossible — needs future)
                in future

Linux uses: Clock-like approximation with active/inactive lists.
```

### Copy-on-Write (COW)

```
fork() doesn't copy memory immediately!

Parent and child share the same physical pages (read-only).
When either writes → page fault → kernel copies just that page.

Result: fork() is nearly free if child calls exec() immediately
(which replaces all pages anyway).
```

## Why It Matters

- Virtual memory protects processes from each other (separate address spaces).
- Understanding paging explains why adding RAM helps and why programs use swap.
- Page faults, TLB misses, and COW directly affect performance.
