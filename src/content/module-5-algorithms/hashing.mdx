## Hook — Beyond Hash Tables

You learned hash tables in the data structures module. Now let's go deeper: How do we design good hash functions? What is universal hashing? How does hashing power algorithms beyond key-value lookup — like Rabin-Karp string matching and consistent hashing in distributed systems?

## Hash Function Design

A good hash function must:
1. **Distribute uniformly** — minimize collisions
2. **Be deterministic** — same input always gives same output
3. **Be fast** — O(1) to compute
4. **Avalanche** — small input change causes large output change

### Division Method

```
h(k) = k mod m

Choose m = prime not close to a power of 2.
  m = 7:   h(14)=0, h(15)=1, h(16)=2 — good distribution
  m = 8:   h(14)=6, h(15)=7, h(16)=0 — only looks at last 3 bits!
```

### Multiplication Method

```
h(k) = floor(m × (k × A mod 1))

A = (√5 - 1)/2 ≈ 0.618... (golden ratio — proven good distribution)
Works well regardless of m (m doesn't need to be prime).
```

### String Hashing (Polynomial Rolling Hash)

```
h("cat") = ('c' × p² + 'a' × p¹ + 't' × p⁰) mod m
         = (99 × 961 + 97 × 31 + 116) mod m

p = 31 (common choice for lowercase letters)
m = 10⁹ + 7 (large prime)
```

## See It Work — Universal Hashing

A **universal hash family** is a set of hash functions where for any two distinct keys, the probability of collision is at most 1/m:

```
Family: h_a,b(k) = ((a × k + b) mod p) mod m
  where p is prime > universe, a ∈ [1, p-1], b ∈ [0, p-1]

Pick a random function from this family.
Guarantee: For any two keys x ≠ y, Pr[h(x) = h(y)] ≤ 1/m

This prevents adversarial inputs from causing worst-case collisions!
```

## How It Actually Works — Hash-Based Algorithms

### Rabin-Karp String Matching — O(n + m) expected

Find pattern P in text T using rolling hash:

```
Text:    "abcdeabc"
Pattern: "abc" (hash = h("abc") = 42)

Slide window and compare hashes:
  h("abc") = 42 → match hash! verify → MATCH at position 0
  h("bcd") = 57 → no match
  h("cde") = 73 → no match
  h("dea") = 89 → no match
  h("eab") = 31 → no match
  h("abc") = 42 → match hash! verify → MATCH at position 5

Rolling hash update: O(1) per position
  h("bcd") = (h("abc") - 'a' × p²) × p + 'd'
```

### Bloom Filters

A space-efficient probabilistic set:

```
Insert: Hash element with k hash functions, set those k bits.
Query:  Check k bits. All set → "probably yes". Any unset → "definitely no".

False positives possible. False negatives impossible.
Space: ~10 bits per element for 1% false positive rate.

Used in: spell checkers, network routers, databases (avoid disk reads).
```

### Consistent Hashing (Distributed Systems)

```
Traditional: server = hash(key) % num_servers
  Problem: Adding/removing a server reshuffles ALL keys!

Consistent hashing: Arrange servers on a ring (0 to 2³²).
  hash(key) → walk clockwise to nearest server.
  Adding a server only moves keys from ONE neighbor.

Used by: Amazon DynamoDB, Apache Cassandra, CDNs.
```

## Why It Matters

- Hash function quality determines hash table performance — bad functions cause O(n) degradation.
- Universal hashing provides provable guarantees against adversarial inputs.
- Hash-based algorithms (Rabin-Karp, Bloom filters, consistent hashing) are used in systems at massive scale.
