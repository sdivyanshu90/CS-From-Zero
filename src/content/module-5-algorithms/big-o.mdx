## Hook — How Fast Does It Grow?

If your algorithm takes 1 second for 1,000 items, how long for 1,000,000? The answer depends entirely on the algorithm's **growth rate**. Big-O notation is the universal language for describing this — every CS course, every textbook, every interview uses it. Master Big-O and you can instantly evaluate whether an algorithm is fast enough.

## What Is Big-O Notation?

**Big-O** describes the upper bound on how an algorithm's time (or space) grows as input size n approaches infinity, ignoring constants and lower-order terms.

```
T(n) = 3n² + 5n + 100

As n gets large, 3n² dominates:
  n = 10:    300 + 50 + 100 = 450       (n² term is 67%)
  n = 100:   30000 + 500 + 100 = 30600  (n² term is 98%)
  n = 1000:  3000000 + 5000 + 100        (n² term is 99.8%)

So T(n) = O(n²) — we drop the constant 3 and lower terms 5n + 100.
```

### Common Complexity Classes

```
Big-O         Name            Example              n=1M ops
─────         ────            ───────              ─────────
O(1)          Constant        Array access         1
O(log n)      Logarithmic     Binary search        20
O(n)          Linear          Linear search        1,000,000
O(n log n)    Linearithmic    Merge sort           20,000,000
O(n²)         Quadratic       Bubble sort          1,000,000,000,000
O(2ⁿ)         Exponential     All subsets          Unfathomable
O(n!)         Factorial       All permutations     Even more unfathomable
```

### Growth Rate Visualization

```
n      O(1)  O(log n)  O(n)   O(n log n)  O(n²)     O(2ⁿ)
10     1     3         10     33          100       1,024
100    1     7         100    664         10,000    1.3 × 10³⁰
1000   1     10        1000   9,966       1,000,000 Too large
```

## See It Work — Analyzing Code

### O(1) — Constant

```c
int first = arr[0];  // One operation regardless of array size
```

### O(n) — Linear

```c
int sum = 0;
for (int i = 0; i < n; i++) {
    sum += arr[i];  // n iterations → O(n)
}
```

### O(n²) — Quadratic

```c
for (int i = 0; i < n; i++) {
    for (int j = 0; j < n; j++) {
        // Runs n × n = n² times → O(n²)
    }
}
```

### O(log n) — Logarithmic

```c
int i = n;
while (i > 1) {
    i = i / 2;  // How many times can you halve n? log₂(n) times.
}
```

### Nested with Different Bounds

```c
for (int i = 0; i < n; i++) {         // O(n)
    for (int j = 0; j < m; j++) {     // O(m)
        // ...                         // O(n × m)
    }
}
```

## How It Actually Works — Formal Definitions

```
Big-O (upper bound):   f(n) = O(g(n)) if f(n) ≤ c·g(n) for large n
Big-Ω (lower bound):   f(n) = Ω(g(n)) if f(n) ≥ c·g(n) for large n
Big-Θ (tight bound):   f(n) = Θ(g(n)) if f(n) = O(g(n)) AND f(n) = Ω(g(n))

Example: Merge sort is Θ(n log n) — always takes n log n, best and worst case.
         Quicksort is O(n²) worst case, but Θ(n log n) average case.
```

### Best, Average, Worst Case

```
Linear search for value in array of size n:
  Best case:    O(1) — found at first position
  Average case: O(n/2) = O(n) — found halfway through
  Worst case:   O(n) — found at last position or not at all

We usually care about worst case (guaranteed upper bound).
```

### Space Complexity

```
Algorithm          Time        Space      Why
─────────          ────        ─────      ─────
Merge sort         O(n log n)  O(n)       Needs extra array for merging
Quicksort          O(n log n)  O(log n)   Stack frames for recursion
Heapsort           O(n log n)  O(1)       In-place
Binary search      O(log n)    O(1)       Iterative version
DFS on graph       O(V + E)    O(V)       Stack for visited nodes
```

### Amortized Analysis

```
Dynamic array append:
  Most appends: O(1) — just write to next slot
  Occasional:   O(n) — resize and copy everything

  Over n appends: total work ≈ 3n → amortized O(1) per append
  (Because resizes at sizes 1, 2, 4, 8, ..., n cost 1+2+4+...+n ≈ 2n)
```

## Why It Matters

- Big-O lets you predict performance before writing code.
- It reveals when an algorithm will hit a wall: O(n²) on 1 million items = 1 trillion operations.
- It's the foundation of every algorithm discussion in interviews, papers, and documentation.
- Choosing the right complexity class matters more than any optimization within that class.
